    - また、チームでフロントエンドに強い人材が少なかったため、自分がリードして API コールライブラリの呼び出しの設計・Storybook 8 の書き方の共有・バリデーションライブラリの使い方の共有を行った

    - MySQL のクエリで使用したプレースホルダーが最大値6万件を超え、エラーが発生した
      - 本番環境では大規模なデータが存在し、それに対応できていなかった
      - 複数回に分割して実行することにより対応した
    - 様々な制約に対応した HTML メールの作成
      - HTML メールには、各種メールエンジンの対応状況からほとんどモダンな HTML 機能を使用できず、基本的に div タグではなく table タグで実装するなどの様々な制約が存在する
      - また、リッチな HTML メールの作成が要求されており、フロントエンドでのプレビュー表示の要件もあったことから、フロントエンドで文面を組み立て、バックエンドへ送信する設計にした
      - 設計・開発にあたり、自分が HTML メールを記述する際の制約・注意事項をドキュメントにまとめて開発メンバーへ周知、フロントエンドでも table タグで作成した Div や Span などの基礎コンポーネントを用意し、それらを利用する構造にした
  - 開発環境の整備
    - タスクランナーの Makefile から [Task](https://taskfile.dev/) への移行を主導した
      - 説明会により課題間の共有を行ったが、エンジニア3チーム14人全員の合意形成に至らず、最終的には導入まで持ち込めなかった
      - 参考: [Makefile から Taskfile へ移行するモチベーションの言語化 - Zenn](https://zenn.dev/hiroto_ohira/articles/8b4b8c8396fcd1)
    - デプロイ手順が整理されていないことにより、作業時間の長時間化・俗人化・不安が発生していたため、手順書を整備した
    - ビジネス的な制約により月次リリースを採用しており、リリースフローが複雑になっているところ、フィーチャーフラグの採用を推進した
      - 3チームあったこともあり、バックマージによるデグレが多々発生しており、その度にチーム間で調整するコストが発生していた
      - 最終的には [DevCycle](https://www.devcycle.com/) を採用した
  - バックエンドのリアーキテクチャ
    - バックエンドをリプレイスする話が持ち上がり、自分が立候補し選抜された
    - バックエンドのタスクランナー・ディレクトリ構造の決定・ライブラリの選定・エラーハンドリングやログの方針・CI の構築など、プロジェクトの初期構築を自分がメインで担当した
      - 比較ライブラリをリストアップしドキュメントに列挙、特徴をまとめつつチームで議論して決定した
      - 多機能さから、タスクランナーには [Task](https://taskfile.dev/) を採用した
      - Go の思想を優先し ORM を採用しない、[bluele/factory-go](https://github.com/bluele/factory-go), [go-testfixtures/testfixtures](https://github.com/go-testfixtures/testfixtures), [gotestyourself/gotestsum](https://github.com/gotestyourself/gotestsum) の採用などの意思決定を行なった
    - 設計手法には Clean Architecture とドメイン駆動設計 (DDD) を採用し、途中から参加したメンバーと一緒に議論しながら導入を進めた
      - モデルのテストデータを疎結合にするため、自分がファクトリーの導入を提案し、最終的にビルダーパターンのファクトリが導入された
      - コードの見通しの良さを保つため、converter 層を導入し、モデルの変換ロジックを一箇所にまとめた
      - 一覧画面などでクエリ効率を優先するため、自分が CQRS の導入を提案し、複雑になりすぎない範囲でコマンド用とクエリ用のモデルを分離した
      - オブジェクトの更新処理で「〇〇が変更されていたら、外部通信する」処理の実装をどうするか議論し、最終的にドメインイベントを採用した
        - id をもとにデータを取得し、ドメインモデルのメソッドで Update するとき、「〇〇が変更されていること」はドミインモデルでしか判定できないが、「外部通信する」ことはドメインモデルの責務外であり、何かしらの連携が必要だった
        - ドメインサービスに切り出すことも考えたが、更新処理は壊れやすくバグインパクトも大きいため、ドメインモデルのメソッドで行いたかった
        - 折衷案としてドメインイベントを採用した
