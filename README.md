### 求人プラットフォームの開発 (2024年9月 - 現在)

- 概要
  - 求人プラットフォームシステムのフロントエンド・バックエンド開発
- 使用技術
  - JavaScript / TypeScript / React / Next.js
  - Go / Gin
  - MySQL
  - Docker / Terraform
  - AWS EC2 / ECS / Fargate / RDS / Aurora
- チーム構成
  - PO1名、ScM1名、プログラマー4名
- 開発手法
  - スクラム
- 業務内容
  - 新規チームでの開発
    - 自分含め全チームメンバーが新規参画の新しいチームだった
    - 誰もプロダクトのドメイン知識やお作法を知らない状態から、一丸となって開発にあたった
  - かなり本格的なスクラムによる開発
    - スクラムイベントで1日が終わる日があるほどイベントに時間を割いて開発を回していた
    - 1週間スプリントであった
  - 新規機能の開発
    - 求人プラットフォームのオプション商品の開発を行った
    - フロントエンドでは、オプション商品には種類がある特性上、後ほど拡張可能な構造にする必要があり、その設計を自分が行い導入した
    - また、チームでフロントエンドに強い人材が少なかったため、自分がリードして API コールライブラリの呼び出しの設計・Storybookの導入・バリデーションライブラリの理解とオンボーディングを行った
    - バックエンドでは、2層によるかなり密結合したアーキテクチャであり、実質的にテストを書くことが不可能だった
    - オプション商品には計算ロジックが複雑であるという特性があるため、無理やり interface を導入し、一部だけテスト可能な状態を作り出した
    - 上記において、interface による依存性の逆転やテスト方針の決定は自分が担当した
  - 開発環境の整備
    - タスクランナーの Makefile から [Task](https://taskfile.dev/) への移行を主導した
      - 参考: [Makefile から Taskfile へ移行するモチベーションの言語化 - Zenn](https://zenn.dev/hiroto_ohira/articles/8b4b8c8396fcd1)
    - デプロイ手順が整理されていないことにより、作業時間の長時間化・俗人化・不安が発生していたため、手順書を整備した
    - ビジネス的な制約により月次リリースを採用しており、リリースフローが複雑になっているところ、フィーチャーフラグの採用を推進した
      - 3チームあったこともあり、バックマージによるデグレが多々発生しており、その度にチーム間で調整するコストが発生していた
      - 最終的には [DevCycle](https://www.devcycle.com/) を採用した
  - バックエンドのリアーキテクチャ
    - 滞在中にバックエンドをリプレイスする話が持ち上がり、自分が抜擢された
    - バックエンドのタスクランナー・ディレクトリ構造の決定・ライブラリの選定・エラーハンドリングやログの方針・CI など、プロジェクトの初期構築を自分がメインで担当した
    - 設計手法には Clean Architecture とドメイン駆動設計 (DDD) を採用し、途中から参加したメンバーと一緒に議論しながら導入を進めた
- 学んだこと
  - 新規機能の開発
